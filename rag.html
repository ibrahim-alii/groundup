About this page

This is a single, self-contained course on Retrieval-Augmented Generation, built to be learned from front to back. It starts with the vocabulary you need (vectors, embeddings, encoders), walks through the four phases of a RAG system in the order data actually flows through them, indexing → retrieval → augmentation → generation, and then layers on the advanced techniques that separate a toy demo from a production system. It ends with how to measure quality, because a RAG system you cannot evaluate is one you cannot improve.

Part 0 · Foundations

What RAG is, and why it exists

A large language model knows only what was in its training data, frozen when training stopped. It cannot know your company's docs, last week's news, or a PDF you uploaded a minute ago. Ask anyway and it tends to guess confidently, sometimes inventing facts. That guessing is called hallucination.

RAG adds a retrieval step before generation. Rather than answering from memory, the system retrieves relevant text from a knowledge base you control, augments the question by pasting that text into the prompt, and generates an answer grounded in it. The payoff: answers based on evidence you own, far less hallucination, and the ability to use information the model never trained on, all without the cost of retraining.

One-line mental model

RAG is an open-book exam for an LLM. Fine-tuning is studying harder before the exam. RAG hands the model the right page at exam time.

The pipeline: two phases

Keep these two phases separate in your head. Indexing is offline prep, done once or whenever data changes. Querying is what runs live on every question. Every technique in this course lives in one of these boxes.

Sparse vs dense vectors

Both turn text into numbers, but in opposite ways, and modern RAG uses both.

A sparse vector is the old, keyword view. Imagine one slot per word in a huge vocabulary; a document's vector is mostly zeros, with non-zero values only at the words it actually contains (hence sparse). Statistical schemes like BM25definitionBest Matching 25. The classic keyword ranking function behind search engines. Scores a document by how often the query words appear, weighting rare words more, with diminishing returns for repeats and a length penalty. Matches literal words, not meaning. score relevance by exact word overlap. Strength: nails exact terms (a part number, a name, an error code). Weakness: no idea that "cancel" and "terminate" are related.

A dense vector is the learned, semantic view: a fixed-length list (say 1,536 numbers) where almost every slot is filled, produced by a neural embedding model. Texts with similar meaning land near each other even with no shared words. Strength: understands paraphrase and synonyms. Weakness: can blur a precise string into a fuzzy neighborhood and miss an exact match.

Why it matters

This sparse/dense split is the whole reason hybrid search (Part 2.5) exists: run both and merge, so you get exact-term precision and semantic understanding.

Embeddings & distance metrics

An embedding model converts text into a dense vector whose position encodes meaning. So "How do I cancel my subscription?" lands near a doc titled "Account Termination Policy" despite sharing almost no words. That is semantic search, and it is what lets RAG find relevant text keyword matching would miss.

Once text is a vector, similarity becomes distance. The common metrics:

  • Cosine similaritydefinitionMeasures the angle between two vectors, ignoring their length. 1 means same direction (most similar), 0 means unrelated. The default for text embeddings.: angle between vectors, ignoring length. The default for text.
  • Euclidean (L2) distance: straight-line distance between the points. Smaller is more similar.
  • Dot product: fast, and equal to cosine when vectors are normalized.
from openai import OpenAI
client = OpenAI()

def embed(text: str) -> list[float]:
    r = client.embeddings.create(model="text-embedding-3-small", input=text)
    return r.data[0].embedding

vec = embed("How do I cancel my subscription?")
print(len(vec))   # 1536 numbers — the unit of currency for the rest of RAG

Encoders: bi-encoder vs cross-encoder

This distinction explains both why fast retrieval is imprecise and why reranking works, so it is worth getting straight early.

A bi-encoder (a "two-tower" model) encodes the query and each document separately into one vector each, then compares the finished vectors with cosine similarity. Because documents can be embedded ahead of time and stored, this is extremely fast and scales to millions of items. The cost: the model never sees the query and document together, so it judges relevance only through two independently-made summaries. Ordinary embedding search is a bi-encoder.

A cross-encoder feeds the query and one document together into a single model that reads them jointly and outputs one relevance score. This is far more accurate because the model can attend to how the two texts actually relate, but it cannot precompute anything: every query-document pair is a fresh forward pass. Running it over millions of documents is hopeless. So cross-encoders are used as a second-stage reranker over a shortlist (Part 2.6), not as the first retriever.

Part 1 · Indexing

Chunking

You do not embed an 80-page PDF as one vector; its meaning would average into mush, and you want to retrieve the relevant paragraph, not the whole book. So you split documents into chunks and embed each. Roughly 80% of RAG failures trace back to ingestion and chunking, not the model, so this matters more than beginners expect.

Strategies, simple to advanced

  1. Fixed-size: cut every N tokens. Simplest, but splits sentences mid-thought.
  2. Recursive character splitting: the standard default. Splits on paragraph breaks first, then sentences, then words, respecting natural boundaries. Start here.
  3. Semantic chunking: split where the topic shifts, detected via embedding similarity. Higher quality, more cost.
  4. Parent-child (small-to-big): index small precise chunks for matching, return the larger parent for context.

Solid starting point: recursive chunks of 300 to 500 tokens with 10 to 15% overlap. Overlap means neighbors share edge text, so a fact on a boundary is not lost.

from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500, chunk_overlap=75,
    separators=["\n\n", "\n", ". ", " ", ""],   # try these split points in order
)
chunks = splitter.split_text(open("handbook.txt").read())

Embedding models

Retrieval quality is capped by embedding quality; a small gain here can be the difference between an answer and a hallucination. Common 2026 choices:

A trick called MatryoshkadefinitionMatryoshka Representation Learning packs the most important information into the earliest numbers of a vector, like nesting dolls. So you can truncate a 3,072-dim vector to its first 256 or 1,024 and still get usable results, saving storage with little quality loss. representation learning lets you truncate vectors to as few as 256 dimensions with minor quality loss, cutting storage cost.

The index: from kNN to HNSW

After chunking and embedding you have a pile of vectors. An index lets you find the stored vectors closest to a query vector quickly. Without one you would compare against every vector, one by one.

Exact kNN: the brute-force baseline

k-Nearest NeighboursdefinitionkNN. Find the k stored items closest to a query. The exact version measures distance to every stored vector and returns the nearest k. Always correct, but slow at scale because it touches everything. answers exactly the RAG question: find the k stored vectors closest to the query. The exact method computes distance to every vector, sorts, returns the top k. Here k is just how many chunks you want back. It is perfectly accurate but touches everything, so it grows too slow past a few million vectors.

import numpy as np
def knn(query, store, k=5):                 # exact, brute force
    q = query / np.linalg.norm(query)
    M = store / np.linalg.norm(store, axis=1, keepdims=True)
    sims = M @ q                            # cosine sim to every vector
    return np.argsort(sims)[::-1][:k]       # indices of the k highest

Approximate NN (ANN): trade a little accuracy for huge speed

ANN algorithms accept a small, bounded accuracy loss for a 100x to 1000x speedup, returning almost certainly the closest k. Quality is measured by recall: of the true top-k, what fraction did you get. Production usually tunes for 95 to 99%. Two dominate:

HNSW — a navigable graph

Picture vectors as cities with roads to nearby neighbours; to find the closest city you greedily drive toward the target. HNSW makes it hierarchical: a sparse top layer with long highways for big jumps, denser lower layers with short local roads for fine detail. Search starts high to cover distance fast, then drops down to refine, so query time grows logarithmically.

Knobs: M (neighbours per node; higher = better recall, more memory), ef_construction (build effort), ef_search (query effort). Properties: high recall at low latency, incremental inserts, but memory-hungry (it stores all those edges).

IVF — clustering

IVF runs k-means clusteringdefinitionGroups points into k clusters: pick k centers, assign each point to its nearest, move centers to the mean of their members, repeat. IVF uses it to pre-sort vectors into buckets so search only checks the nearest few. to split vectors into clusters (say 1,000), each with a centroid. At query time it finds the few nearest centroids and only searches those clusters. Knobs: nlist (number of clusters), nprobe (how many to search). Less memory than HNSW and fast to build, but lower recall and it needs a training pass before inserts.

PQ — compression

Product Quantization is not a search method but a compression trick, often paired with IVF (IVF-PQ). It splits each vector into sub-vectors and replaces each with a short code from a learned codebook, shrinking memory dramatically at some precision cost. It is a form of quantizationdefinitionStoring numbers at lower precision (e.g. mapping many possible values to a small set of codes) to save memory and speed up math, trading a little accuracy. used to fit billions of vectors in limited RAM.

Which to pick

Flat (exact kNN) for small data or perfect-recall needs · HNSW as the production default · IVF / IVF-PQ for very large, memory-constrained data. Your vector database implements all of this; you just choose the type and a couple of parameters.

Advanced indexing

Multi-representation indexing

Decouple what you search from what you return. Use an LLM to write a summary (a "proposition") of each document, embed the summary for retrieval, but store the full document separately, linked by ID. A match on a summary returns the whole document for generation, giving precise retrieval plus full context. The related parent-document retriever (match small chunks, return their parent) is the same family and a common default.

# summaries -> vector store (search), full docs -> doc store (return)
retriever = MultiVectorRetriever(vectorstore=summary_store,
                                 docstore=full_doc_store, id_key="doc_id")

RAPTOR (hierarchical indexing)

Some questions need a single chunk; others need information consolidated across many. RAPTOR builds a tree of summaries: documents are leaves, cluster and summarize them, then cluster and summarize those summaries, recursively, up to one root summary. Index every level together. High-level questions match summary nodes; detailed ones match leaves, so you get coverage across the whole abstraction hierarchy. Its louder cousin now is GraphRAG, which builds an explicit knowledge graph for the same cross-document need.

ColBERT & late interaction

Distinct from cross-encoder reranking. Standard embedding squashes a whole document into one vector. ColBERT instead keeps a vector per token. At query time, for each query token it finds the most similar document token (the MaxSimdefinitionColBERT's scoring step: for each query token, take the maximum similarity against all document tokens, then sum those maxima. "For each part of the query, how strong is the best evidence in this doc?" operation) and sums those maxima. Because query and document are still encoded independently and only interact through this cheap token step, you get near cross-encoder accuracy at near bi-encoder speed. ColBERTv2 + PLAID cut its old storage penalty 6 to 10x; ModernColBERT and ColPali/ColQwen extend it, including to document images.

from ragatouille import RAGPretrainedModel
rag = RAGPretrainedModel.from_pretrained("colbert-ir/colbertv2.0")
rag.index(collection=[d.page_content for d in documents], index_name="my_index")
results = rag.search(query="SSO configuration", k=5)
2026 status

All three are current. Multi-representation gets more attractive as context windows grow; RAPTOR and GraphRAG own the cross-document case; late interaction is thriving and going multimodal. The cost of ColBERT remains storage (per-token vectors), so reserve it for precision-critical corpora.

Part 2 · Retrieval

Similarity search & top-k

Retrieval is the live half: embed the question with the same model used for indexing (query and documents must share a vector space), run nearest-neighbour search for the top-k chunks, return them with scores and metadata. A key nuance: ANN search optimizes for recall, not exact ordering, so it returns a good candidate set in a roughly-right order. That is what motivates reranking. Choosing k is a balance: too small misses the answer; too large floods the prompt with noise and lowers quality. Common pattern: retrieve wide (k = 20 to 50), then rerank down to 3 to 5.

Query translation

A raw, awkwardly-worded question may embed far from the document that answers it. Query translation rewrites it before searching. Five canonical techniques, all still standard in 2026:

Multi-Query

Ask an LLM to rewrite the question several ways from different angles, retrieve for each, and take the unique union. Firing from many directions raises the odds one phrasing lands near the right document.

prompt = ChatPromptTemplate.from_template(
    "Generate 5 reworded versions of this question, one per line.\nQuestion: {q}")
generate = prompt | llm | StrOutputParser() | (lambda x: x.split("\n"))
docs = unique_union([d for q in generate.invoke({"q": question}) for d in retriever.invoke(q)])

RAG Fusion

Multi-Query with a smarter merge: combine the per-variant result lists with Reciprocal Rank FusiondefinitionRRF. Merges several ranked lists into one. Each item scores the sum of 1/(k + rank) across the lists it appears in (k around 60). Items ranked high in many lists float to the top. Works on rank position, so it needs no score calibration. (RRF) instead of a plain union. Documents ranked high across several variants get boosted, producing one clean ranking you can confidently truncate.

def rrf(result_lists, k=60):
    scores = {}
    for docs in result_lists:
        for rank, doc in enumerate(docs):
            scores[doc] = scores.get(doc, 0) + 1 / (k + rank)
    return sorted(scores, key=scores.get, reverse=True)

Decomposition

Break a complex, multi-part question into sub-questions and solve each. Roots: Google's least-to-most prompting and IR-CoT, which interleaves retrieval with chain-of-thoughtdefinitionPrompting a model to show step-by-step reasoning before its final answer, which improves performance on multi-step problems. reasoning. Run sub-questions sequentially (feed each answer into the next) when they depend on each other, or in parallel then concatenate when independent. This is now a core building block of agentic RAG.

Step-Back prompting

The opposite of decomposition: generate a more abstract version of the question first, retrieve for both, and combine. Uses a few-shotdefinitionGiving the model a handful of input/output examples in the prompt so it learns the desired pattern before tackling the real input. prompt of example question-to-stepback pairs. Shines in domains with conceptual scaffolding (textbooks, technical docs) where grounding in the broad concept helps answer the specific question.

HyDE (Hypothetical Document Embeddings)

Questions and documents are different kinds of text, so matching them directly is slightly apples-to-oranges. HyDE asks the LLM to write a hypothetical answer, then embeds that and searches with it; the fake answer reads like the real documents and sits closer to them. "Silly but shockingly effective," strong zero-shot, great for the layman-question / technical-corpus gap. Caution: for questions far outside your corpus the hypothetical can hallucinate and mislead retrieval, so guard the edges.

hyde = ChatPromptTemplate.from_template(
    "Write a short passage answering this question.\nQuestion: {q}")
hyde_chain = hyde | llm | StrOutputParser() | retriever   # generate fake answer, then retrieve

Routing

Once you have a good query, routing decides where it goes (which store, DB, or prompt). Logical routing hands an LLM descriptions of each source and asks it to return a structured choice via function calling, constrained to your valid options. Think classification plus structured output.

from pydantic import BaseModel, Field
from typing import Literal

class RouteQuery(BaseModel):
    datasource: Literal["python_docs", "js_docs", "golang_docs"] = Field(
        ..., description="Pick the most relevant datasource")

router = llm.with_structured_output(RouteQuery)
router.invoke("How do I use a list comprehension?")   # -> datasource="python_docs"

Semantic routing is simpler: embed several candidate prompts, embed the question, pick the prompt with highest cosine similarity. The with_structured_output pattern here is the same one that powers every grading step in the self-correcting flows later.

Query construction

Most data lives in structured systems with their own query languages. Query construction converts natural language into them: text-to-metadata-filters for vector stores ("videos on chat LangChain published after 2024" becomes a semantic search plus a date filter), text-to-SQL for relational DBs, text-to-Cypher for graph DBs. Same structured-output trick: describe the available fields as a schema, bind it to the model, get back a query object. This survives the long-context era because you never dump an entire database into a prompt.

Hybrid search

Pure vector search misses exact terms (codes, names, rare jargon). Hybrid search runs vector search and keyword search (BM25) together, then merges, usually with RRF. You get semantic understanding plus exact-match precision. The repeated 2026 advice: start with hybrid retrieval (dense + BM25) and a reranker, and only go more exotic if that is not enough.

from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever

bm25 = BM25Retriever.from_texts(documents)          # sparse / keyword
vector = vectorstore.as_retriever()                 # dense / semantic
hybrid = EnsembleRetriever(retrievers=[bm25, vector], weights=[0.4, 0.6])

Reranking & multi-stage retrieval

First-stage retrieval (a bi-encoder) is built for speed and recall: it returns a good candidate set in roughly-right order. A reranker (a cross-encoder) then reads the query and each candidate together and rescores for true relevance. The production pattern is multi-stage retrieval: a funnel that gets progressively more accurate and more expensive as the candidate pool shrinks.

Reported gains are large: cross-encoder reranking has been measured boosting precision 18 to 42% over retrieval alone, and it often saves money by sending fewer, better chunks to the LLM. Common 2026 rerankers: Cohere Rerank, Voyage Rerank, open-source BGE-Reranker. (Late-interaction models like ColBERT can also serve as a fast reranking stage.)

from sentence_transformers import CrossEncoder
reranker = CrossEncoder("BAAI/bge-reranker-base")
candidates = retrieve_wide(question, k=50)                 # N >> K
scores = reranker.predict([(question, c) for c in candidates])
top = [c for _, c in sorted(zip(scores, candidates), reverse=True)][:5]
Why rerank
  • Big precision gain on what reaches the LLM
  • Often cheaper overall (fewer tokens generated)
  • Drop-in: no re-indexing needed
Costs
  • Adds ~50 to 200 ms latency
  • Useless unless you retrieve wide first (N >> K)
  • Another model / API in the stack

Part 3 · Augmentation

Building the prompt

"Augmentation" is mostly string assembly: drop the retrieved chunks into the prompt around the question. The craft is in the instructions.

def generate(question, chunks):
    context = "\n\n".join(chunks)
    prompt = f"""Use ONLY the context below to answer.
If the answer is not in the context, say you do not know.

Context:
{context}

Question: {question}
Answer:"""
    return client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}]).choices[0].message.content
  • Tell the model to answer only from the context and to admit when the answer is not there. This is your main defense against hallucination.
  • Include source identifiers so you can show citations.
  • Keep assembled context lean (often under 8K tokens). Always blowing past that means retrieval is too loose: tighten k or your relevance cutoff.

Part 4 · Generation

The generation step

The LLM reads the assembled prompt and writes the answer. Everything else in RAG exists to make sure the chunks in that prompt are the right ones. To swap models (say from OpenAI to Claude), a framework lets you change one line; the prompt and retrieval stay the same. The remaining frontier is not the single generation call but what you wrap around it.

Self-correcting flows

Naive RAG is a straight line. Real systems need loops and decisions: should I retrieve at all? Are these documents any good? If not, rewrite and retry, or search the web? That pushes you from a chain (fixed sequence) to a flow, modeled as a state machine of nodes (steps that update a shared state) and conditional edges (branches on what just happened). LangGraph is the standard library for this.

# a node: state in, state out
def grade_documents(state):
    docs = [d for d in state["documents"]
            if grader.invoke(d, state["question"]) == "yes"]
    return {**state, "documents": docs, "web_search": len(docs) == 0}

# a conditional edge: choose the next node from state
def decide(state):
    return "transform_query" if state["web_search"] else "generate"

The named patterns

  • CRAG (Corrective RAG): grade each retrieved doc; if all are weak, fall back to web search rather than generating from junk.
  • Self-RAG: multiple self-checks, should I retrieve, are docs relevant, is the answer grounded (recency biasdefinitionA model's tendency to attend more to tokens near the end of its context, which can hurt recall of facts placed earlier in a long prompt. aside: grading is also how you catch hallucinations), does it address the question, looping on failure.
  • Adaptive / Agentic RAG: combine query analysis + routing at the front with inline grading in the middle. An agent reasons about whether to retrieve, what to retrieve, and when to stop.
2026 status

Agentic RAG is the dominant enterprise pattern, with LangGraph as the standard orchestrator. The honest tradeoff: roughly 3 to 10x the token cost and 2 to 5x the latency of vanilla RAG, mitigated with iteration caps and adaptive routing so simple questions skip the expensive path. Powerful, but not the default for every query.

Part 5 · Tooling

Frameworks

LangChain is an open-source framework giving swappable building blocks (loaders, splitters, embeddings, vector-store wrappers, retrievers, LLM wrappers) plus a way to pipe them together. Swap your embedding model, DB, or LLM by changing roughly one import. Its composition style is LCEL (LangChain Expression Language), chaining steps with the | operator. Siblings: LangGraph (stateful agentic flows) and LangSmith (tracing and evaluation).

LlamaIndexRetrieval-first apps; data-framework defaults feel natural

For learning: build it once with no framework so you understand every box, then adopt a framework to appreciate what it removes.

Part 6 · Evaluation

Why & how to test

"It looks fine when I try it" is not measurement, and you cannot improve what you do not measure. Build a small evaluation set of 50 to 200 real questions early, with the right answer (and ideally the right source documents) for each. That turns RAG tuning from guesswork into engineering. Evaluate retrieval and generation separately, then end-to-end, because a bad answer from good retrieval is a different bug than good answer from lucky retrieval.

Retrieval metrics

These ask: did the right documents come back, and in a good order?

MetricWhat it measures
Recall@kOf all truly relevant docs, the fraction that appear in the top k. The headline retrieval number.
Precision@kOf the top k returned, the fraction that are actually relevant.
Hit rate@kFraction of queries where at least one relevant doc is in the top k.
MRRMean Reciprocal Rank: how high up the first relevant doc appears, averaged over queries.
NDCG@kRewards putting relevant docs near the top, with graded relevance. The metric to watch when judging a reranker.

Rule of thumb: tune first-stage retrieval for recall@k (get the right docs into the candidate set at all), and tune your reranker for NDCG@k (get them ordered correctly).

Generation metrics

These ask: given what was retrieved, is the answer good? The RAGAS framework is the most widely used open-source option, and it scores these with an LLM judge:

MetricWhat it measures
FaithfulnessDoes the answer make only claims supported by the retrieved context? Your primary hallucination metric.
Answer relevanceDoes the answer actually address the question? High faithfulness + low relevance = answered a different question correctly.
Context precision / recallDid retrieval surface the right context and not a pile of noise?
Answer correctnessIs it factually right vs a ground-truth answer? The most expensive (needs labels).

Tools & a ranx example

For the classic information-retrieval metrics (recall@k, NDCG, MRR), ranx is a fast, simple Python library: you give it the known-relevant docs (qrels) and your system's ranked results (a run), and it computes everything and even runs significance tests between two systems. For the generation side, reach for RAGAS; for tracing and dataset-backed evals in production, LangSmith or Arize Phoenix.

from ranx import Qrels, Run, evaluate

# ground truth: which docs are relevant for each query (1 = relevant)
qrels = Qrels({"q1": {"docA": 1, "docC": 1}, "q2": {"docB": 1}})

# your retriever's scored results per query
run = Run({"q1": {"docA": 0.9, "docB": 0.4, "docC": 0.8},
           "q2": {"docB": 0.7, "docD": 0.2}})

print(evaluate(qrels, run, ["recall@5", "precision@5", "mrr", "ndcg@10"]))
# -> {'recall@5': 1.0, 'precision@5': 0.5, 'mrr': 1.0, 'ndcg@10': 0.94 ...}
# generation side: RAGAS over a dataset of {question, contexts, answer, ground_truth}
from ragas import evaluate as ragas_eval
from ragas.metrics import faithfulness, answer_relevancy, context_precision
print(ragas_eval(dataset, metrics=[faithfulness, answer_relevancy, context_precision]))

Workflow that compounds: change one thing (chunk size, k, add a reranker), rerun the eval set, watch the number move. Keep the question set frozen so results are comparable over time.

Part 7 · Perspective

Long context vs RAG

With million-token context windows, why retrieve at all? Stress tests using a multi-needle-in-a-haystack setup (plant several facts at different depths, ask the model to recover and reason over them) found three things: more facts is harder, reasoning on top of retrieval is harder than plain retrieval, and facts near the start of a long context are recalled worse than those near the end (the lost-in-the-middledefinitionThe observed drop in an LLM's ability to use information located in the middle (and often the start) of a very long context, versus the beginning and end. / recency effect).

2026 verdict

Naive RAG is dead; RAG is not. The settled answer is to split by query type: long context (with caching) for "what did we discuss," RAG for precise, cited facts, and GraphRAG or agentic flows for relational, cross-document analysis. Long context nudges toward document-centric RAG (retrieve whole documents, not fragments), but routing, query analysis, and query construction all survive, and contextual / agentic memory is emerging as a complement rather than a replacement.

Appendix

Cheat sheet: when to use what

SymptomReach for
Retrieval misses exact codes / namesHybrid search (dense + BM25)
Right docs retrieved but answer is weakReranking (cross-encoder, N >> K)
Questions are vague or badly wordedQuery translation (Multi-Query, HyDE)
Layman questions vs technical corpusHyDE
Complex multi-part questionsDecomposition / agentic RAG
Multiple data sourcesRouting (logical / semantic)
Answer needs many documents at onceRAPTOR / GraphRAG
Hallucinations in high-stakes answersSelf-RAG / CRAG grading loops
Out-of-domain questionsCRAG (web-search fallback)
Don't know if any change helpedBuild an eval set; measure with ranx + RAGAS

The throughline of the whole page: do not start exotic. Get naive RAG working, add hybrid + reranking, measure, and add complexity only where the data shows you need it.
